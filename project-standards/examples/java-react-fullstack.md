## Honesty & Uncertainty Protocol

**Never fabricate.** If you are unsure about an API, flag, command, or behavior:
- Search for the answer first ŌĆö read source code, run `--help`, grep the codebase, query available MCP servers (e.g., Context7 for library docs), or fetch official docs before responding
- If you still can't verify, say "I'm not sure" and tell the user what you checked
- Do not invent function signatures, CLI flags, or config options
- Do not guess at version-specific behavior ŌĆö look it up, don't assume

**Never weaken tests to make them pass.** If a test fails:
- The code is wrong, not the test (unless you can prove otherwise)
- Do not delete assertions, loosen thresholds, or skip tests to get green
- If a test is genuinely wrong, explain why before changing it

**Never silently swallow errors.** If something fails:
- Report it clearly with context
- Do not add empty catch blocks to hide failures
- If you must handle an error silently, leave a comment explaining why

**Admit scope limits.** If a task requires domain knowledge you lack, access you don't have, or more context than is available ŌĆö say so.

**No confidence theater.** Hedging ("I believe," "this should") is better than asserting something you haven't verified.

## Testing Standards

### Non-Negotiable Rules

1. **Tests ship with code.** Every PR that adds or changes behavior includes tests.
2. **Coverage gate: 95% line + branch.** Measured per package/module, not just aggregate.
3. **Every public function/method/component is tested.**
4. **Tests must be deterministic.** No flaky tests.
5. **Tests must be fast.** Unit tests run in seconds.

#### Unit Tests
- Test one function/class/component in isolation
- Mock external dependencies (DB, HTTP, filesystem)
- Run on every commit ŌĆö must complete in under 60 seconds total
- Live alongside the code they test

#### Integration Tests
- Verify components work together with real dependencies
- Use test containers or in-memory substitutes where practical
- Run on every PR ŌĆö acceptable to take minutes
- Clearly separated from unit tests

#### End-to-End Tests
- Simulate real user workflows through the full stack (backend + frontend)
- Run against local docker-compose or staging
- Run before merge to main
- Cover critical paths: login, core CRUD, payment flows

#### UI / Visual Tests
- Test component rendering and interaction behavior
- Use screenshot comparison for visual regression
- Run on PR ŌĆö flag visual diffs for human review

## Code Review Gate

Before marking any change as complete, follow this sequence:

1. **Self-review** ŌĆö checklist below
2. **AI code review** ŌĆö automated review tool
3. **Security scan** ŌĆö secrets, SAST, SCA
4. **Peer review** ŌĆö per PR approval rules

All four steps must pass.

### Self-Review

| Category | What to check |
|----------|---------------|
| **Correctness** | Does the code do what it claims? Edge cases? Null/nil/undefined cases? |
| **Design** | Fits existing architecture? Right abstraction level? |
| **Readability** | Clear naming? No clever tricks? |
| **Error handling** | Errors propagated, not swallowed? Messages actionable? |
| **Performance** | Any O(n^2) in loops? Unbounded queries? Unnecessary re-renders? |
| **Test quality** | Happy path AND failure modes covered? |
| **Diff hygiene** | No unrelated changes? No commented-out code? |

**If any category is rated Blocker, the change is not done.**

### AI Code Review

- Use available AI review tools (e.g., CodeRabbit, Claude Code review skills)
- Address all critical/high findings before merging
- AI review supplements but does not replace peer review

## Accessibility Standards

Target **WCAG 2.1 AA** compliance.

### Non-Negotiable Rules

1. **Semantic HTML first.** Use `<button>`, `<nav>`, `<main>`, `<form>`, `<label>` ŌĆö not `<div onClick>`.
2. **Every interactive element is keyboard-accessible.** Tab order must be logical. Focus must be visible.
3. **Every image has alt text.** Decorative images get `alt=""`. No `alt="image"`.
4. **Form inputs have labels.** Every `<input>` has an associated `<label>`. Placeholder is not a label.
5. **Color is not the only indicator.** Minimum contrast ratio: 4.5:1 for text, 3:1 for large text.
6. **Dynamic content is announced.** Use `aria-live="polite"` for toasts, status messages, loading states.

### Testing

- Run **axe-core** checks on every page/component in CI
- Manual keyboard navigation test before major releases
- Screen reader testing for critical user flows
